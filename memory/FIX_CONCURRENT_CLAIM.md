# 修复并发领取身份重复问题

## 问题描述

**症状**：当多个用户同时点击"领取身份"时，会出现重复的身份号码。例如3个人，有两个人领取到1号（重复），一个人领取到3号，2号没有人领取。

**根本原因**：
1. 数据库同步延迟：多个用户几乎同时创建领取记录时，在验证阶段可能看不到对方的记录
2. 验证时间窗口不足：原来的150ms等待时间不够，导致并发冲突检测不可靠
3. 缺少预检查：在创建记录前没有先检查座位是否已被占用

## 修复方案

### 1. 增强 `tryCreateClaimAtomically` 函数 (utils/cloud.js)

**关键改进**：

#### a) 添加预检查机制
```javascript
// 在创建记录前先检查座位是否已被占用
const existingClaims = await db.collection(CLAIMS_COLLECTION)
    .where({
        planId: planId,
        seatIndex: seatIndex
    })
    .get()

if (existingClaims.data && existingClaims.data.length > 0) {
    // 座位已被占用，直接返回冲突
    return { success: false, conflict: true }
}
```

#### b) 增加等待时间和验证次数
- 初始等待时间：150ms → **200ms**
- 验证次数：3次 → **4次**
- 每次验证间隔：50ms → **60ms**
- 需要连续2次查询结果一致才算稳定

#### c) 增强日志输出
- 打印每次验证的结果
- 打印所有竞争者的详细信息
- 使用 ✅ 和 ❌ 标记成功/失败

#### d) 赢家清理输家记录
```javascript
if (amIWinner) {
    // 清理其他输家的记录（帮助清理数据库）
    const losers = allClaims.data.slice(1)
    for (const loser of losers) {
        await db.collection(CLAIMS_COLLECTION).doc(loser._id).remove()
    }
}
```

### 2. 优化重试逻辑 (pages/view/view.vue)

**关键改进**：

#### a) 增加最大重试次数
- 10次 → **15次**

#### b) 指数退避策略
```javascript
// 冲突重试延迟：100ms + retry * 50ms (最大500ms)
const delay = Math.min(100 + retry * 50, 500)

// 网络错误延迟：300ms + retry * 100ms (最大1000ms)
const delay = Math.min(300 + retry * 100, 1000)
```

#### c) 增强日志输出
- 打印每次尝试的次数
- 打印剩余空位数和已占用数
- 打印随机选择的座位信息

## 测试方法

### 测试场景1：3人同时领取（高并发）

1. 创建一个3人游戏（3个平民，0个卧底，0个白板）
2. 使用3台手机或3个微信账号
3. 同时（在1秒内）点击"领取身份"按钮
4. **预期结果**：
   - 3个人分别领取到1号、2号、3号（不重复）
   - 控制台日志显示可能有冲突和重试，但最终成功

### 测试场景2：5人同时领取（极端并发）

1. 创建一个5人游戏
2. 使用5台手机或5个微信账号
3. 同时（在1秒内）点击"领取身份"按钮
4. **预期结果**：
   - 5个人分别领取到不同的号码（1-5号）
   - 控制台日志可能显示多次冲突和重试

### 测试场景3：重复领取

1. 用户A领取身份后
2. 用户A再次点击"查看我的身份"
3. **预期结果**：
   - 显示相同的身份号码
   - 不会创建新的领取记录

### 测试场景4：所有座位已满

1. 创建一个3人游戏
2. 3个用户都已领取身份
3. 第4个用户尝试领取
4. **预期结果**：
   - 提示"所有座位已被占用"
   - 不会创建新的领取记录

## 关键技术点

### 1. 数据库索引建议

为了提高查询性能，建议在云数据库的 `claims` 集合上创建以下索引：

```json
{
  "planId": 1,
  "seatIndex": 1
}
```

### 2. 并发控制原理

使用"乐观锁"策略：
1. 先创建记录（乐观地认为不会冲突）
2. 创建后立即验证（检查是否有冲突）
3. 如果有冲突，通过时间戳和记录ID决定赢家
4. 输家删除自己的记录并重试

### 3. 时间戳排序

使用服务器时间戳 `db.serverDate()` 而不是客户端时间，避免客户端时间不同步的问题。

## 注意事项

1. **数据库权限**：确保 `claims` 集合的权限设置为"所有用户可读、所有用户可写"
2. **网络延迟**：在网络较慢的环境下，可能需要更多的重试次数
3. **日志监控**：建议在生产环境中监控冲突率，如果冲突率过高，可以进一步优化
